# Groupyライクなウィンドウタブ化アプリ 要件定義書 (MVP版)

## 1. はじめに
本ドキュメントは、市場にある「Stardock Groupy」のようなウィンドウをタブ化するアプリケーションの自作に向けた要件をまとめたものです。
最初の段階（MVP: Minimum Viable Product）では、**「任意のデスクトップアプリケーション（複数開いているウィンドウ）を一つのウィンドウでタブ化して管理しやすくする」** ことを目的として要件を定義します。

---

## 2. コア要件（第一段階 / MVP）

初期段階として最低限実装すべき機能は以下の通りです。

### 2.1. 対象アプリケーションの取得
- 実行中のすべてのウィンドウプロセスを検知・リストアップできること。
- タスクバーに表示されるような、通常のトップレベルウィンドウを対象とする。

### 2.2. 基本GUI（コンテナウィンドウ）
- Chromeブラウザのような、画面上部にタブバーを備えた洗練されたUI。
- 従来のリスト選択型ではなく、直感的にタブが並ぶデザインとする。

### 2.3. ウィンドウの統合と分離（ドラッグ＆ドロップ動作）
- **統合（アタッチ）**: 外部ウィンドウのタイトルバーをドラッグし、本アプリのタブバー領域にドロップすることで、瞬時にタブとして取り込まれる。
- **分離（デタッチ）**: 本アプリ上でタブをドラッグして外へスワイプ（ドロップ）することで、タブから外れ、元の独立したウィンドウに戻る。

### 2.4. 位置とサイズの同期
- コンテナウィンドウを移動・リサイズした際、現在アクティブな対象ウィンドウも完全に追従して移動・リサイズされること。
- ※OSネイティブのようなシームレスさを出すため、コンテナのクライアント領域に対象アプリケーションがピッタリ収まるように制御する。

---

## 3. 将来的な拡張要件（Groupyを参考にした高度な機能）

MVPが完成した後、本格的なGroupyに近い使い勝手を実現するための機能拡張案です。

- **自動グループ化 (Auto-grouping):** 同じアプリケーション（例: 複数のエクスプローラーやWord文書）が新しく起動した際、自動的に既存のタブグループに追加するルール設定。
- **グループ（セッション）の保存:** タブの構成（どのアプリを開いているか）を保存し、次回起動時に一括で起動・グループ化を復元する機能。
- **タブの視覚的カスタマイズ:** 視認性を高めるためのタブの色分けや、ハイライト、Fluent Designなどの適用。

## 4. 技術的なアプローチ案（Windows環境）

任意のウィンドウをまとめるためには、Windows APIを利用した高度なウィンドウ制御が必要です。主なアプローチは以下の2つが考えられます。

### アプローチA: ウィンドウの埋め込み (`SetParent` API の利用)
対象のウィンドウを、自作アプリのウィンドウの子要素（Child）として設定してしまう方法です。
- **メリット:** 一度子要素にしてしまえば、移動などの同期がOS側で自動的に処理されるため実装の基本部分が楽。
- **デメリット:** ChromeやVSCodeのような独自のレンダリングエンジンを持つ最新アプリや、複雑なウィンドウメッセージを持つアプリの場合、フォーカスの喪失、リサイズ時の描画乱れなどが発生しやすい。

### アプローチB: 同期型オーバーレイ（Zオーダーと座標の同期）
対象ソフトは独立したウィンドウのまま維持し、自作の「タブバーのみのウィンドウ」を対象ソフトのタイトルバー上部に覆い被せる（または対象ソフトに常に追従させる）方法です。
- **メリット:** `SetParent`による予期せぬ不具合を回避しやすく、対象アプリがクラッシュするリスクが低い。
- **デメリット:** 移動時やリサイズ時に数ミリ秒の遅延（ラグ）が発生しやすく、コンテナと中身がズレて見えることがある。フック技術などを用いて完璧な同期を行うには工夫が必要。

### 推奨技術スタック
Windows API（Win32 API）を頻繁に呼び出す必要があるため、以下のような技術が適しています。
- **C# (.NET / WPF / WinForms):** `user32.dll` のP/Invokeが容易であり、UI構築も生産性が高いため、本プロジェクトでのメイン開発言語として推奨。
- **C++:** Windows APIをネイティブに扱えるため、パフォーマンスや低レイヤーのフック制御（将来的な高度化）において、必要に応じて部分利用（DLL等）を検討する。

## 5. 開発・運用フロー

ソースコードのバージョン管理およびプロジェクト全体の進行管理には **GitHub** を利用して開発を進める。

### 5.1. バージョン管理
- **Git / GitHub:** 全ソースコードはGitでバージョン管理し、GitHub上のリポジトリを中央のソースとする。
- **ブランチ戦略:** 「GitHub Flow」などシンプルな運用を基本とし、機能の追加や修正の度にブランチを作成・Push後、Pull Request（PR）経由でメインブランチにマージする。

### 5.2. プロジェクト管理とCI/CD（推奨）
- **タスク管理:** GitHub Issues または GitHub Projects（カンバン）を利用し、MVPで実装する機能群やバグトラッキングを行う。
- **自動化 (GitHub Actions):** ソースコードのPushやPR作成時に、自動でビルド検証や基本テストが行われる環境（CI環境）の構築を将来的に検討する。
- **リリース:** リリースビルドの生成や配布パッケージ（インストーラー等）の作成もGitHub Releasesと連携して自動化することで、運用の手間を削減する。
